### H1. Photographer 覆盖同名真实玩家，并在移除时把真实玩家从 `playersByName` 一并删除 [L§2.5，加深]

**文件**：
- `leaves-server/minecraft-patches/features/0144-Replay-Mod-API.patch:361-362`（placeNewPhotographer 注册）
- `leaves-server/minecraft-patches/features/0144-Replay-Mod-API.patch:485-491`（removePhotographer 清理）

**证据**：patch 在 `placeNewPhotographer` 里无条件 put：
```java
this.playersByName.put(player.getScoreboardName().toLowerCase(java.util.Locale.ROOT), player);
this.playersByUUID.put(player.getUUID(), player);
```
`removePhotographer` 则：
```java
this.playersByName.remove(entityplayer.getScoreboardName().toLowerCase(java.util.Locale.ROOT));
...
if (entityplayer1 == entityplayer) {
    this.playersByUUID.remove(uuid);
}
```

**问题**：
1. 如果 photographer 名字 == 某真实 player 名字（常见：录制者自己就叫 `MyName`），真实玩家在 `playersByName` 中的条目被覆盖为 photographer → `server.getPlayerByName("MyName")` 返回 photographer。
2. **更严重**：photographer 移除时 `playersByName.remove(name)` 无差别删除，真实玩家的条目被一起抹掉。此后 `getPlayerByName("MyName")` 永远返回 null，直到玩家重登。
3. UUID 分支因为有 `if (entityplayer1 == entityplayer)` 守卫，不会误删真实玩家，但名字分支没有对应守卫。

**影响**：其他插件（经济/聊天/权限）按名字查玩家会在录制期间得到错误对象，录制后得到 null。

**LIMITATIONS.md §2.5 只提到"覆盖"未提"移除时殃及"**，这是比 L§2.5 更深的问题。

**建议**：`placeNewPhotographer` 里跳过对 `playersByName`/`playersByUUID` 的写入（photographer 不应参与常规名字查找），或在 `removePhotographer` 里用"只在是自己时移除"守卫（与 UUID 分支对齐）。

### H2. 0123-Fix 补丁把 `BuddingAmethystBlock.getResetPushReaction()` 的 `@Override` 移除，丢失编译期契约检查 **NEW**

**文件**：
- `leaves-server/minecraft-patches/features/0123-Fix-latent-compile-errors-in-rebased-patches.patch:204-210`
- 原 patch：`leaves-server/minecraft-patches/features/0039-Movable-Budding-Amethyst.patch:17`
- 基类定义：`leaves-server/minecraft-patches/features/0124-Leaves-Utils.patch:85-94`（Block.java 加 `public PushReaction getResetPushReaction()`）
- 调用点：`leaves-server/minecraft-patches/features/0124-Leaves-Utils.patch:108`（BlockBehaviour.getPistonPushReaction 调 `this.getBlock().getResetPushReaction()`）

**证据**：0039 原有 `@Override`，0123 把它删掉：
```
-    @Override
     @org.jetbrains.annotations.Nullable
     public net.minecraft.world.level.material.PushReaction getResetPushReaction() {
```

**问题**：所有 patch 应用完成后，`Block.getResetPushReaction()` 基方法确实存在（来自 0124）。BuddingAmethystBlock 的这个方法在 final code 里**事实上仍在覆盖**父类方法，移除 `@Override` 不改变 Java 语义——但丢失了编译器"该方法必须覆盖基类"的契约检查。

**风险**：未来 Paper/Leaves 改名或改签名（譬如加形参、重命名），这里静悄悄脱钩，`BuddingAmethystBlock` 变成无效死方法，`movable-budding-amethyst` 配置悄悄失效。

**建议**：把 `@Override` 加回来。如果重加会有编译错（疑为 jspecify `@Nullable` 位置导致 javac 认为覆盖不匹配），应对齐两处的 nullable 注解位置而不是删注解。

### H3. BotStatsCounter 删了 `parseLocal` override，新的 `parse(DataFixer, JsonElement)` 没有对应 override [L§2.2，加深]

**文件**：`leaves-server/src/main/java/org/leavesmc/leaves/bot/BotStatsCounter.java:28-30`

**证据**：
```java
// Leaves - Paper 26.1: parseLocal removed from ServerStatsCounter; the JSON parsing moved to parse(DataFixer, JsonElement)
// Keeping method gone: our bot stats are stubbed (never saved/loaded), so the old override had no behavior to preserve
```
注释自己承认"JSON parsing moved to parse(DataFixer, JsonElement)"，但**没 override 这个新方法**。

**问题**：
- 若父类 `ServerStatsCounter` 构造或其他路径在某个时机调 `parse(DataFixer, JsonElement)`（原本 `parseLocal` 的替代点），并尝试读 `UNKOWN_FILE = Path.of("BOT_STATS_REMOVE_THIS")`（文件不存在），行为未知：可能捕获并记 WARN、也可能抛 IOException。

**LIMITATIONS.md §2.2 把它标为"可能日志噪音"**，严重性被低估：如果父类在读 bot 进入时必调 parse，bot 登入可能失败（取决于异常传播）。

**建议**：补一个空 override：
```java
@Override
public void parse(DataFixer fixer, JsonElement json) { /* bot stats not persisted */ }
```
（签名需与 26.1 版本对齐，以实际代码为准）

### M1. `Recorder.forceDayTime` 以 0-24000 范围值当 `ClockNetworkState.totalTicks` 使用 [L§2.3]

**文件**：`leaves-server/src/main/java/org/leavesmc/leaves/replay/Recorder.java:204-214`

**代码**：
```java
packet = new ClientboundSetTimePacket(
    packet1.gameTime(),
    net.minecraft.util.Util.mapValues(
        packet1.clockUpdates(),
        state -> new net.minecraft.world.clock.ClockNetworkState(recorderOption.forceDayTime, 0.0F, 0.0F)
    )
);
```

**问题**：
- 旧 3 参构造 `(dayTime, totalTicks, bool)` 里第一个就是 time-of-day 值
- 新 `ClockNetworkState` 的第一个构造参数**不一定**是 time-of-day；结合 Paper 26.1 的 `WorldClock` 体系，更可能是"绝对 totalTicks"
- 若字段真实语义是 totalTicks：客户端显示时间会基于 `forceDayTime % DAY_LENGTH` 看似对（当 forceDayTime < 24000）但各维度的独立 WorldClock 全部被覆盖成相同 tick，Nether/End 的时间行为可能错乱

**建议**：重放一段录制，对比客户端显示时间是否为预期的"正午 6000"等。若错位，按 §LIMITATIONS 建议重算 `forceDayTime + (originalTotalTicks - originalTotalTicks % 24000)`。

### M3. 0145 Old-Block-Remove-Behaviour：`ObserverBlock` 和 `RedstoneTorchBlock` 未调 super.onRemove() **NEW**（低危）

**文件**：
- `leaves-server/minecraft-patches/features/0145-Old-Block-remove-behaviour.patch:487-494`（Observer）
- 同 patch `:541-547`（RedstoneTorch）

**代码**（Observer）：
```java
@Override
protected void onRemove(BlockState state, Level level, BlockPos pos, BlockState newState, boolean isMoving) {
    if (!state.is(newState.getBlock())) {
        if (state.getValue(POWERED) && level.getBlockTicks().hasScheduledTick(pos, this)) {
            this.updateNeighborsInFront(level, pos, state.setValue(POWERED, Boolean.FALSE));
        }
    }
    // 缺 super.onRemove 调用
}
```

**分析**：LIMITATIONS 未提及。审查后**降级到"非 bug"**：
1. BlockBehaviour 基类的 super.onRemove 只做两件事：`AsyncCatcher.catchOp("block remove")` + 如果有 BlockEntity 就 `level.removeBlockEntity(pos)`。
2. ObserverBlock 与 RedstoneTorchBlock **均无 BlockEntity**，所以 BE 清理无意义。
3. 1.21.1- vanilla 原始代码里这两个块也**没有调 super**（本 patch 的目的就是"恢复 1.21.1- 行为"），所以是忠实复刻。

**唯一副作用**：AsyncCatcher 不会触发，理论上如果有异步代码意外触发这两个方块的 onRemove，这里不会抓住；但 vanilla 原代码也没抓，属于可接受的 1.21.1- 行为复刻。

**建议**：保持现状，加注释说明"此处未调 super 是有意为之（复刻 1.21.1- vanilla）"会减少未来误修的概率。

### M4. `LeavesMinecraftSessionService` 用自建 MinecraftClient，未覆盖父类对同一 client 的其他内部用法 [L§2.4，加深]

**文件**：`leaves-server/src/main/java/org/leavesmc/leaves/profile/LeavesMinecraftSessionService.java:36-41, 80`

**证据**：
```java
private final MinecraftClient leavesClient; // mirror of YggdrasilMinecraftSessionService#client (now private)
...
this.leavesClient = MinecraftClient.unauthenticated(authenticationService);
...
// 在 hasJoinedServer 里：
final HasJoinedMinecraftServerResponse response = this.leavesClient.get(url, ...);
```

**问题**：
1. `leavesClient` 与父类 `YggdrasilMinecraftSessionService.client` 是**两个独立实例**；若 Mojang authlib 将来在父类 client 上加 header/timeout 配置、加 metrics/tracing 钩子，`leavesClient` 不会继承。

**建议**：PR 前最好通过反射取父类 `client` 字段，或向 Mojang authlib 申请 protected getter。短期内保留。

### M5. AbstractVillager 的 `SimpleContainer` 匿名类 setChanged override 在 **构造期**就能触发 PCA 同步 **NEW**

**文件**：`leaves-server/minecraft-patches/features/0137-Fix-PCA-addListener-and-REI-display-API-mismatch.patch:37-52`

**代码**（final）：
```java
private final SimpleContainer inventory = new SimpleContainer(8, (CraftAbstractVillager) this.getBukkitEntity()) {
    @Override
    public void setChanged() {
        super.setChanged();
        if (!AbstractVillager.this.level().isClientSide() && org.leavesmc.leaves.LeavesConfig.protocol.pca.enable) {
            org.leavesmc.leaves.protocol.PcaSyncProtocol.syncEntityToClient(AbstractVillager.this);
        }
    }
};
```

**原始（26.1 前）实现**：在构造函数里判断非客户端后 `addListener`：
```java
if (!this.level().isClientSide()) {
    this.inventory.addListener(...);
}
```

**潜在差异**：
- 原始：listener 在构造函数中安装；如果在 listener 安装前 `SimpleContainer` 构造内部触发了 setChanged（理论上 26.1 的 SimpleContainer 初始状态不触发 setChanged，但未验证），listener 不会跑
- 新：匿名子类从 **第一次调 setChanged 起**就会尝试同步；若构造期间 super.setChanged() 就被调用（尤其是 SimpleContainer 父类 constructor 里 fill/init items），会在 AbstractVillager 对象还未完全构造好时调 `AbstractVillager.this.level()`。此时 `level` 字段通常在 super(EntityType, Level) 里已赋值，**应当安全**，但依赖了构造顺序。

**建议**：运行时观察一下 villager trade 开始/结束、交易刷新时是否有双重同步或空指针。低风险，但迁移时值得确认一次。

AbstractHorse 同理（`createInventory` 在 attach tag 时调用，此时 this 已完全构造，没有风险）。
